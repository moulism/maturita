Indexy a rychlost dotazů

⏱ ~15 minut
📝 Potítko – napiš si na papír
  • Index = rejstřík v knize — rychlé vyhledání bez procházení všech stránek
  • Zrychluje SELECT | Zpomaluje INSERT, UPDATE, DELETE
  • Typy: Clustered (1x na tabulku, fyzické řazení) | Non-clustered | Unique
  • EXPLAIN / Execution Plan → Table Scan = špatné | Index Seek = dobré
  • Příliš mnoho indexů = pomalý zápis + zbytečné místo na disku

Co je index a jak funguje

Index v databázi funguje jako věcný rejstřík na konci tlusté knihy. Bez rejstříku musíte přečíst každou stránku (Table Scan — extrémně pomalé). S rejstříkem se podíváte na písmeno S, zjistíte, že SQL je na straně 150, a otevřete knihu rovnou tam (Index Seek — bleskové).

Business příklad: E-shop s milionem klientů. Hledám zákazníka podle e-mailu. S indexem = milisekunda. Bez indexu = server na několik vteřin zamrzne.

Kdy index pomůže a kdy škodí

OperaceVliv indexuProč
SELECT (čtení)✅ Dramaticky zrychlujeDatabáze jde přímo na správné místo
INSERT (vkládání)⚠️ ZpomalujeIndex se musí aktualizovat u každého nového záznamu
UPDATE (úprava)⚠️ ZpomalujeStarý záznam se z indexu odstraní, nový se přidá
DELETE (mazání)⚠️ ZpomalujeZáznam musí být odstraněn i z indexu

Typy indexů

Clustered Index
Fyzicky řadí data na disku v pořadí indexu. Každá tabulka může mít jen JEDEN. Typicky na ID (primárním klíči). Data jsou fyzicky seřazená jako stránky v knize.
Non-clustered Index
Vytvoří oddělený rejstřík ukazující na fyzická data. Může jich být více. Používá se pro sloupce, podle kterých se hodně vyhledává (e-mail, příjmení).
Unique Index
Zaručuje rychlé vyhledání a navíc zakáže duplicitní hodnoty. Business: nasazuje se na sloupec E-mail — dva zákazníci nesmějí mít stejnou adresu.
Composite Index
Index na více sloupcích najednou. CREATE INDEX ON Objednavky(zakaznik_id, datum) — efektivní pokud hledáme vždy podle obou sloupců.

Jak zjistit, zda dotaz index používá

Použijeme příkaz EXPLAIN (MySQL) nebo Execution Plan (MS SQL). Grafický strom ukáže, jak databáze dotaz pochopila:

Index Seek = databáze šla přímo do indexu ✅ Výborně!
Index Scan = prošla celý index — trochu lepší než Table Scan
Table Scan = prošla celou tabulku řádek po řádku ❌ Problém — přidej index!

Business: slouží pro kontrolu, proč se měsíční report generuje 5 minut místo 5 vteřin.

⚠️ Chytáky a záludné otázky
  • „Přidám index na všechny sloupce — je to dobré?" — Špatně! Každý index fyzicky zabírá místo na disku (někdy více než samotná data) a zpomaluje každý INSERT/UPDATE/DELETE. Více indexů = pomalejší zápis.
  • „Kdy NEPOUŽÍVAT index?" — Na malých tabulkách (databáze stejně projde vše). Na sloupcích s málo unikátními hodnotami (pohlaví M/F — index nepomůže). Na tabulkách s extrémně vysokou frekvencí zápisů (IoT senzory).
  • „Co je to Table Scan?" — Databáze prochází každý řádek tabulky jeden po druhém, jako kdybyste hledali slovo v knize bez rejstříku a četli každou stránku.
💡 Tip na 15 minut — říkej komisi „Index není zadarmo — je to vždy kompromis. Pro archivační systém, kam se zřídka nahrávají smlouvy ale tisíce lidí v nich vyhledává, nasadím indexů hodně. Pro tabulku IoT senzorů, které zapisují teplotu tisíckrát za vteřinu, nesmím dát skoro žádný — neustálý přepočet indexu by databázi zablokoval."
